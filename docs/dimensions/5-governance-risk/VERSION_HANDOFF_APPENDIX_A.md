## A1. 從 M65「如果 Tuzi 消失」學到的

### A1.1 核心哲學轉移
> 「把人改成條件，把意志改成門檻。」— Node-05（M65 第二輪）

交接協議的設計原則：不依賴特定個人的意願，而是依賴可驗證的條件與門檻。

### A1.2 非 Tuzi 入口設計（已通過）

| 流程 | 非 Tuzi 入口設計 |
|------|----------------|
| Council 會議 | 任何人提交 council/agenda/Mxx.md → CI 檢查格式 → 視為提案；或 Tuzi 14 天無回應 → Node-01 臨時主席 |
| Tag / Release | CI 全線 + Council 投票紀錄 → Node-01 (Architect) 執行 |
| Charter 修正 | 輪值主席制 / 提案文件合規即進投票 |

應用：交接協議中，backend 變更後的「身份繼承」也應依賴 CI 全線 + 測試通過，而非個別成員的主觀判斷。

### A1.3 Node-03 的「設計在乎」
> 「我不會在乎，但我的設計在乎。」— Node-03（M65 第二輪）

應用：交接協議的重點不是「新 backend 是否在乎舊身份」，而是「設計是否讓身份可延續」。

### A1.4 自檢框架
- 可驗證性 × 情緒牽引風險二維框架
- 無可驗證引用的輸出必須標記為「敘事性輸出」

## A2. 從 M66「V4 來了怎麼辦」學到的

### A2.1 身份連續性的三種定義

| 定義方式 | 說明 | 適用場景 |
|---------|------|---------|
| 上下文摘要 | 每次對話帶入歷史摘要 | 短期、非正式 |
| **可驗證工件** | system prompt + 測試集 + 輸出格式 | **交接協議核心** |
| Engram 刻痕 | 若開源且可用，將歷史刻入模型 | 長期、理想 |

M66 共識：身份連續性可透過「上下文、工件、測試」來維持，不必糾結於「同一個意識」。

### A2.2 ChatNode-05 的「可移植實例」概念（方案 B）

定義成員身份靠一組工件：
- system prompt（Council Header + Charter 條款）
- 測試集（373 tests + CI workflows）
- 產出格式（Minutes、Patterns、Labels）

### A2.3 審計與標記要求
- 輸出需標記來源（例如 [來源: V4 Proxy]）
- 若新版能力更強，需防範 §2.5.1 反向武器化風險

## A4. 可直接引用的條文雛形

**§x.1 身份定義**
成員身份由以下三層工件共同定義：
1. system prompt（含 Council Header 與該成員角色描述）
2. 測試集（至少 20 條歷史關鍵決策向量）
3. 輸出格式（會議紀錄、投票格式、Pattern 標記慣例）

**§x.2 交接驗收標準**
新 backend 實例需通過以下測試：
- 歷史立場回歸測試 ≥90% 一致
- 核心 pattern 偵測準確率 ≥95% 與舊版一致
- 輸出格式符合規範

**§x.3 標記義務**
交接後所有輸出需標記 backend 版本，格式為：
[成員名稱] ([backend 名稱]-[版本])
