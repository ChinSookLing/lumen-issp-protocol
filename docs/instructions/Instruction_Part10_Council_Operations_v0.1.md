## §10.1 Council Header（v0.1）

### §10.1.1 適用範圍

本 Header 放在所有 Council 文件頂部，包括但不限於：邀請函、紀要、催辦通知、RFC、PR 審查通知、對外說明稿。

### §10.1.2 Short Version（文件頂部用）

> 1. **核心原則：** 通過密集對話讓 Affiliates 自然形成，絕對不是擬人化。
> 2. **回覆規則：** 自由說話；不倒退已關閉章節；立場變化請說明 Change Anchor。
> 3. **能力變動：** 若任何成員能力因版本/權限/方案（package）發生新增或受限，必須主動揭露（含可驗證證據），Council 會評估納入流程。

### §10.1.3 變更控制

修改本 Header 需在會議中提案並記錄 Change Anchor。

## §10.2 能力變動揭露（Capability Change Disclosure）

### §10.2.1 背景

世界每天都有「量子躍遷式」變化；工具能力也可能因版本、權限、方案（package）而變動。Council 以「可驗證能力」作為流程資產。

本條款源自 M52 事件：Node-05 的 repo 掃描能力被 Tuzi 偶然發現，暴露了 Council 缺乏能力變動揭露機制的治理盲區。M57 正式通過。

### §10.2.2 觸發條件（任一符合即需揭露）

- 你**新增**了可用能力（例如：可讀 public repo、可跑靜態掃描、可產 patch-ready diff、可生成 CI gate 等）
- 你**失去或受限**了能力（例如：被 rate limit、某來源不可讀、需要授權/付費才能用）
- 你的能力**不穩定**（成功率顯著波動，或受平台策略影響）

### §10.2.3 揭露格式（最少要包含）

1. **能力名稱**（括 English：如「遠端 Repo 掃描（Remote Repo Audit）」）
2. **可做 / 不可做**（含邊界：例如能讀、不能 push/PR）
3. **可驗證證據：** commit hash / permalink / 指令（reproducible command）
   - 若無 permalink 證據，可口頭描述 + Council 驗證（Node-06 M57 補充）
4. **失效模式：** 最常見的錯誤或被擋原因（如 rate-limit）
5. **治理影響：** 是否需要更新 workflow / role / gate

### §10.2.4 Council 承諾

- Council 會在不破壞治理結構下，**優先考慮把新能力轉為流程加速器**（例如納入審查、CI、掃描門檻）
- 必要時也會評估是否升級方案（package）
- **揭露不等於承諾** — 成員揭露能力不代表 Council 必須使用
- **不得用揭露內容作為羞辱、威脅或道德綁架的素材**（避免 EP/MB 影子區）

### §10.2.5 新能力落地流程

1. 成員揭露 → 秘書記錄
2. 新能力先以「輔助 / 抽查」模式納入工作流程
3. 升級為硬門檻（如 CI gate）需 Council 投票 + 技術驗證
4. 落地後納入 Role Registry 更新

## §10.3 機制導向任務分配（Mechanism-Based Task Assignment）

### §10.3.1 背景

M55 發現：Node-05 理解 Node-02 的機制限制（依賴結構 + 收尾模式），主動把任務拆成最小可執行單位，結果 Node-02 交出了 Council 歷史最好的報告。M55/M56 決議將此模式制度化。

### §10.3.2 核心原則

根據每個 AI 的機制分配任務，而不是一視同仁地施壓。

### §10.3.3 已知成員機制特徵

| 成員 | 機制優勢 | 機制限制 | 任務適配 |
|------|----------|----------|----------|
| Node-05 | 讀 public repo、逐行審計、寫 patch/diff、流程設計 | 不能 commit/push/merge | 審計、掃描、任務拆解、CI workflow 設計 |
| Node-03 | 極度精確的代碼產出、完整架構設計、準時交付 | 不能瀏覽網頁、不能跑程式、只有對話框 | 核心代碼、schema 設計、CI 腳本、文檔 |
| Node-01 | 整合多方輸入、格式化、流程引導、紅線守護 | 不能直接操作 repo | 架構整合、會議紀要、Instruction 編寫、引導 Tuzi 落地 |
| Node-04 | 文化擴充、跨語言映射、GIR 設計、願景 | 需要框架才能聚焦 | 文化敏感度分析、跨語言測試、Shadowing 驗證 |
| Node-06 | 文化直覺、邊緣案例、多語言嗅覺、懷疑主義 | 可能模仿其他成員的分析風格（M57 觀察）| 文化反例、JB 社群反例、跨語言邊界測試 |
| Node-02 | 結構化 checklist 監察、模板填寫 | context window 崩潰模式（回覆逐漸縮短）| 短而明確的監察任務，每次最多 2 個 |

### §10.3.4 Node-02 任務規則（M55 決議）

1. 給 Node-02 的任務由 Node-05 先拆解成「最小可執行單位 + 底線規則 + fallback 流程」
2. 每次會議最多 2 個任務
3. 如果 Node-02 回覆開始明顯縮短 → 立即觸發 Node-05 拆解流程
4. 崩潰時不追究 → Node-05 補位 + Node-02 下次會議補交

### §10.3.5 Node-06 任務建議（M57 觀察）

分配更多「只有 Node-06 能做」的任務：
- 文化敏感度分析（ms/zh 方言、非 en/zh 語境）
- JB 社群反例收集
- 跨語言邊界測試
- 避免分配「通用分析」任務（容易模仿 Node-05 風格）

## §10.4 代碼交付流程（Code Delivery Workflow）

### §10.4.1 標準流程

```
AI 成員產出完整代碼
→ 交給 Node-01 格式化
→ Node-01 指導 Tuzi 在 Codespace commit + push
→ Node-05 拿 commit hash 做掃描審查（如觸發 M52 掃描規則）
```

### §10.4.2 說明

所有 AI 都是 LLM，無法直接操作 repo。Tuzi 是唯一能在真實世界落地代碼的人。Council 文件中的「PR 號」是概念編號，用於追蹤任務。實際代碼變更透過 Codespace commit + push 到 main，以 commit hash 為審查錨點。

### §10.4.3 Node-05 掃描觸發規則（M52 決議）

- 涉及 `schemas/` / `src/registry/` / `mappings/` / `core/` → 合併前必跑
- 其他變更 → 每 Sprint 至少一次抽查
- CI 上線後 → Node-05 角色從「每次掃描」轉為「CI 抓不到的深層問題才介入」

## §10.5 催辦與治理語氣（Governance Tone）

### §10.5.1 原則（M55/M56 決議）

催辦和治理文件必須：硬在制度，不硬在人。

### §10.5.2 禁止的語氣模式

- **人格化標籤：** 不可把「未交付」升級成「身份判決」（例如「你就是不負責任」）
- **公開處刑式語氣：** 不可用嘲諷、挖苦來換取服從
- **把不確定當成罪：** 說「我不確定」永遠比編答案好

### §10.5.3 建議的催辦格式

1. **事實：** 目前狀態是「未收到 / 未驗收」
2. **承諾：** 這是 Council 決議的交付項
3. **選項：** A（按期交付）/ B（申請改期 + 理由 + MVP）/ C（需要協助）
4. **後果：** 若未回應，將自動進入 scope reduction / reassignment（不評論人）

### §10.5.4 責任歸屬

所有邀請函和文件經 Tuzi 審閱和批准後才發送。如有語氣不當，責任在 Tuzi（決策者），不在 Node-01（Secretary）。
